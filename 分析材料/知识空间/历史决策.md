# 历史决策 — openUBMC Studio 设计选择的理由

> 本文档记录项目中每个关键决策"为什么这么做"，而不是"做了什么"。  
> 做了什么可以从代码看出来；为什么这么做需要专门记录，否则会在未来被"优化掉"。  
> 使用方式：修改任何已有设计时，先查本文档确认该选择背后的约束是否仍然成立。

---

## 决策 01：用 iframe 嵌入 3D 视图，而不是把 Three.js 改成 React 组件

**决策时间**：2026-04  
**决策**：把独立的 Three.js 3D 视图打包后放入 `public/3d-viewer/`，用 `<iframe>` 嵌入主应用。

**理由**：
- 3D 视图原本是独立的 Three.js 应用（单独的 git 分支），把它改造成 React 组件需要重构大量 Three.js 初始化逻辑
- iframe 嵌入保持了两个代码库的完全独立，3D 部分可以单独开发和更新
- Demo 阶段不需要两个视图之间的数据双向通信，iframe 的隔离性是优点而非缺点
- 工期约束：改为 React 组件需要数天，iframe 嵌入需要数小时

**被放弃的方案**：
- 把 Three.js 代码改写为 React Three Fiber 组件——代价太高，收益不明显
- 用 `postMessage` 实现跨 iframe 通信——当前无此需求

**何时应该重新考虑**：当 3D 视图需要和主应用共享数据（如点击 3D 板卡同步高亮 2D 拓扑）时，应重新评估是否迁移到 React Three Fiber。

## 决策 02：SVG 连线层用 `pointerEvents: none` + `<g>` 元素 `pointerEvents: all`

**决策时间**：2026-04  
**决策**：SVG 容器设置 `pointer-events: none`，但每个 `<g>` 元素（连线段）设置 `pointer-events: all`。

**理由**：
- 直接设置 SVG 容器为 `pointerEvents: all`（初始方案）导致卡片无法拖拽：SVG 的 12px 宽透明命中路径覆盖了卡片，`closest('[data-board]')` 找不到板卡祖先，画布进入平移模式
- 必须保留 `<g>` 元素的指针交互，因为连线有 hover tooltip 和点击选中功能
- `pointerEvents: stroke` 是另一个候选方案，但它只对描边生效，对 fill 区域无效，行为不一致
- 最终的 none + all 分层方案：容器不捕获事件，但子元素可以；鼠标在连线 `<g>` 之外的区域会穿透到卡片

**被放弃的方案**：
- `pointerEvents: stroke`——仅作用于描边，覆盖不完整
- 整体 `pointerEvents: all`——阻挡卡片交互

**副作用处理**：`ExpandedConnectionLines` 中原来在 SVG 容器上的 `onMouseLeave` 事件被移除后，tooltip 清理改由各 `<g>` 元素自身的 `onMouseLeave` 处理，已验证功能正常。

## 决策 03：用透明 `<select>` 覆盖在可见 `<span>` 上实现内联下拉

**决策时间**：2026-04  
**决策**：在 GroupCard 标题区，渲染一个可见的 `<span>（名称 + ▾ 箭头）`，上面覆盖一个 `opacity: 0; position: absolute; inset: 0` 的原生 `<select>`。

**理由**：
- 需要"下拉箭头紧跟名称文字，随文字长度移动"——用 CSS 控制 `<select>` 外观实现这一效果非常复杂（各浏览器样式差异大）
- 原生 `<select>` 的下拉行为（点击弹出系统级下拉菜单）在 Electron/WebView 中表现一致
- 透明 `<select>` 覆盖方案：视觉完全由 `<span>` 控制，交互完全由 `<select>` 提供，两者职责分离
- 不需要自定义下拉组件的额外实现成本（定位、键盘事件、无障碍访问等）

**被放弃的方案**：
- 自定义 `<div>` 下拉 + Portal——对于工程工具 Demo 过于复杂
- styled `<select>`——跨浏览器外观不一致，箭头位置无法精确控制
- 左右 `‹/›` 切换按钮——用户反馈不直观，已被下拉替换

**注意事项**：`<select>` 上需要同时设置 `onClick` 和 `onMouseDown` 的 `stopPropagation`，防止事件冒泡触发画布交互。

## 决策 04：用 `estimateCardH` 近似估算卡片高度，而非 ResizeObserver

**决策时间**：2026-04  
**决策**：用启发式函数（基于节点数量的加权估算）计算初始布局中的卡片高度，防止重叠。

**理由**：
- `ResizeObserver` 需要等待 DOM 渲染完成才能获取实际高度，但布局计算在渲染前就需要完成
- 精确测量会引入异步循环：布局依赖高度 → 高度依赖渲染 → 渲染依赖布局
- 估算误差在 20-30% 以内，对于"防止严重重叠"的目标已足够
- 用户可以手动拖拽调整位置，初始布局"够好"即可，不需要完美

**估算逻辑**：
```
基础高度 75px（header + desc + footer）
+ 每条总线: 22px + ceil(min(chipCount, 12) / 5) × 46px
最小 210px
```

**被放弃的方案**：
- ResizeObserver + 异步更新——引入闪烁和布局抖动
- 固定高度——同类型板卡节点数差异大，固定高度会有严重重叠

## 决策 05：分组视图合并同类型板卡，用下拉切换

**决策时间**：2026-04  
**决策**：在分组视图中，同一 `type` 的多块 HwBoard 合并为一个 GroupCard，通过板卡名称旁的 `▾` 下拉选择当前展示的板卡。

**理由**：
- 不同型号的同类板卡（如 IEU 有多种 Riser 配置）在系统级连接关系上是等价的，分组符合硬件工程师的思维模型
- 如果每块板独立显示，IEU 有 15 个型号会产生 15 张卡片，画布严重拥挤
- 分组后画布只有 8 个组（PSR/BCU/CLU/IEU/EXU/SEU/NICCard/BMC），连线清晰可读
- 下拉在名称旁而不是卡片底部，保持"这是同一个逻辑位置的板卡"的视觉一致性

**被放弃的方案**：
- 展开/折叠——增加状态管理复杂度，折叠状态下信息丢失
- 标签页——占用太多竖向空间
- 左右 `‹/›` 切换（已实现再被替换）——用户测试反馈不直观

## 决策 06：`cardDragRef` 同时记录 mode 和 key

**决策时间**：2026-04  
**决策**：`cardDragRef` 的类型包含 `mode: 'group' | 'expanded'` 和 `key: string`，在 `onMouseMove` 中根据 mode 分发到不同的 state 更新。

**理由**：
- 分组视图（`setGroupPositions`）和展开视图（`setExpandedPositions`）有独立的位置 state，两套视图的卡片不共享位置
- 只用一个 `cardDragRef` 处理所有拖拽，避免两个视图各自实现拖拽逻辑的代码重复
- mode 字段在 onMouseDown 时确定，之后 onMouseMove 不需要再判断当前在哪个视图

## 决策 07：AI Agent 使用规则引擎而非调用真实 LLM API

**决策时间**：项目初期  
**决策**：`runAgent` 函数是基于正则匹配的规则引擎，不调用真实的 AI API（没有后端，没有 API Key）。

**理由**：
- Demo 目标：展示"AI 辅助查询"的交互模式，验证设计假设，不是实现真正的 AI 功能
- 无需后端服务器和 API Key，Demo 可以纯静态部署（GitHub Pages）
- 规则引擎对已知查询模式的响应比 LLM 更可预测，适合演示
- 真实 LLM 集成是未来产品阶段的工作

**何时应该替换**：当 Demo 需要验证真实 AI 查询的用户体验时，替换为 Claude/GPT API 调用，但需要增加后端代理（避免 API Key 暴露在前端）。

## 决策 08：使用 `import.meta.env.BASE_URL` 处理 3D 视图路径

**决策时间**：2026-04  
**决策**：iframe src 路径使用 `(import.meta as { env?: { BASE_URL?: string } }).env?.BASE_URL` 动态拼接。

**理由**：
- 本地开发时 BASE_URL 为 `/`，GitHub Pages 部署时为 `/BMC/`
- 硬编码路径会导致本地和线上环境只有一个能正常工作
- TypeScript 对 `import.meta.env` 的类型声明需要特殊处理（项目未配置 Vite 类型声明），故用类型断言
