# 体验设计师 × AI 工具探索实验 — 工作流总结与沉淀

> 项目背景：openUBMC Studio，一款面向 BMC 硬件工程师的 CSR 拓扑可视化编辑工具  
> 实验角色：体验设计师作为产品主导，AI 作为实现伙伴，全程协作完成从概念到可运行 Demo 的完整链路  
> 文档目的：从体验设计师视角提炼核心思路、可复用模式与可沉淀内容

---

## 一、核心思路：设计师持有"为什么"，AI 执行"怎么做"

这个实验最根本的发现是一个协作分工模型：

```
设计师  →  问题定义 · 判断标准 · 领域知识 · 迭代决策
  ↕
AI      →  代码实现 · 方案推演 · 细节填充 · 错误诊断
```

这不是"让 AI 替代设计师写代码"，而是**设计师通过 AI 把设计意图直接转化为可运行产物**，跳过了传统链路中的设计稿→评审→前端开发→联调的多个环节。

### 1.1 设计师在这个链路中的真正工作

- **语境喂养**：把领域知识（I²C 信号方向、Connector 的两种语义、PSR→IEU 不是直连等）精确告诉 AI，AI 才能生成正确的实现
- **视觉判断**：通过截图反馈"卡片遮挡""箭头位置不对""下拉要在名称旁边"，AI 才知道要改什么
- **架构决策**：决定是用 iframe 嵌入 Three.js 还是改造成 React 组件；决定用分组视图还是展开视图——这些判断 AI 不能独立做出
- **质量把关**：识别"SVG 层挡住了拖拽事件"这类视觉上正确但交互上错误的情况，并能描述现象让 AI 诊断

### 1.2 AI 在这个链路中真正擅长的

- 把设计意图转化为具体实现（bezier 曲线双轨渲染、自适应高度估算、拖拽事件路由）
- 在大型代码库中定位相关位置并做精确修改
- 给出多种方案的权衡（比较 `pointerEvents: stroke` vs 整体透明化）
- 快速实现复杂的 SVG/Canvas 交互，不需要设计师懂每个 API

## 二、工作流模式提炼

### 模式 A：截图驱动的迭代精化

**触发条件**：视觉效果或交互行为与预期不符时  
**流程**：
```
截图 → 描述"哪里不对/我想要什么" → AI 诊断并修改 → 再截图确认
```
**关键技巧**：
- 截图要聚焦到问题区域，而不是全屏截图
- 描述时说"现象"而不是"猜测原因"（"拖拽时画面在移动"比"stopPropagation 没生效"更有效）
- 允许多轮迭代，不要求 AI 一次做对

**本项目案例**：卡片遮挡问题经历了"布局算法→高度估算→指针事件冲突"三个层次的迭代，每一层都依赖截图反馈推动

---

### 模式 B：领域知识注入 → 精准生成

**触发条件**：要实现一个具有领域特殊性的功能时（比如"这根线实际上不是直连的"）  
**流程**：
```
设计师提供领域知识（可以是口头描述、图示、文档片段）
→ AI 将其转化为数据模型/视觉表达
→ 设计师验证语义是否正确
```
**关键技巧**：
- 把隐性知识显性化：不只说"加一条线"，而是说清楚"这根线承载 PCIe 和 I²C 两个信号，但是同一根物理线缆"
- AI 生成的分析文档（如 I²C 连线分析.md）可以作为领域知识的沉淀，下次直接喂给 AI

**本项目案例**：BCU→IEU Hisport 双信号轨渲染、refConnector 桥接概念的可视化表达

---

### 模式 C：渐进式功能叠加

**触发条件**：产品从 0 到 1 的构建阶段  
**流程**：
```
先做最简单可运行版本 → 截图确认视觉方向 → 加一个功能 → 确认 → 再加
```
**关键技巧**：
- 每次只加一个功能点，避免大范围重写带来的不可控性
- "先跑起来"比"一次做完整"更重要
- AI 每次修改后立刻验证，不要积累

**本项目案例**：硬件拓扑视图从静态布局→连线→hover tooltip→拖拽→高度自适应，按功能点逐步叠加

---

### 模式 D：复杂 Bug 的系统性诊断

**触发条件**：一个 Bug 修了还出现，或者行为难以解释时  
**流程**：
```
描述"我期望发生什么 vs 实际发生什么"
→ AI 提出假设 → 验证假设（eval/截图）→ 找到根因 → 修复
```
**本项目典型案例**：
- 拖拽卡片时画布在平移 → 怀疑 stopPropagation → 测试发现 SVG 层 `pointerEvents: all` 导致事件被 SVG 截获，不存在 `[data-board]` 祖先，guard 失效

## 三、可沉淀的内容类型

### 3.1 Skill：可复用的 AI 提示词模板

以下是本项目实践中识别出的高价值 Skill 方向：

#### Skill-01：可交互数据可视化原型生成
```
输入：数据结构描述 + 连接关系语义 + 视觉风格偏好
输出：可运行的 React/SVG 可视化组件，含 hover/click 交互
适用：拓扑图、关系图、流程图、架构图的快速原型
```

#### Skill-02：设计规范提取与一致性修复
```
输入：现有代码库（读取主要组件文件）
输出：提炼出颜色系统、间距规律、字体层级、交互模式
用途：确保新功能与已有设计语言一致；或为新项目建立规范
```

#### Skill-03：截图 → 实现反向工程
```
输入：目标界面截图 + 技术栈说明
输出：还原该界面的代码实现，或分析其交互逻辑
用途：竞品分析、快速仿制参考界面
```

#### Skill-04：领域知识 → 可视化表达映射
```
输入：领域概念描述（可以是口语化的）+ 已有可视化的截图
输出：把新概念融入现有视觉体系的实现方案
用途：专业工具设计（需要将行业概念准确表达为图形）
```

#### Skill-05：交互 Bug 定位与修复
```
输入：截图/录屏描述的现象 + 预期行为
输出：根因分析 + 修复代码
关键：让 AI 先提出假设，设计师判断假设是否符合实际，再实施修复
```

### 3.2 SOP：标准操作流程

#### SOP-01：AI 辅助 Demo 快速开发流程

```
阶段 1 — 准备（1-2h）
  □ 明确 Demo 目标：给谁看？验证什么假设？
  □ 收集领域知识：业务规则、数据结构、专业术语
  □ 确定技术栈：React/Vue/原生 HTML？有无现有代码库？
  □ 收集视觉参考：截图、竞品、设计稿

阶段 2 — 搭骨架（2-4h）
  □ 让 AI 生成最简版本（数据能跑通、布局大致对）
  □ 不追求细节，先验证数据流和交互流是否符合预期
  □ 截图确认整体结构

阶段 3 — 功能叠加（按功能点迭代）
  □ 每次只做一个功能点
  □ 功能完成后立刻截图/操作验证
  □ 不合预期则描述现象让 AI 修改，而非自己修改代码

阶段 4 — 细节打磨
  □ 交互细节：hover、过渡、边界状态
  □ 视觉细节：颜色一致性、间距、字体
  □ 此阶段可以频繁使用截图驱动迭代

阶段 5 — 沉淀
  □ 整理领域知识文档（本次学到的行业概念）
  □ 记录有效的提示词模式
  □ 提交代码到版本库
```

#### SOP-02：复杂 Bug 诊断流程

```
1. 描述现象（不要猜原因）
   → "按住鼠标拖动时，画布在移动，而不是卡片在移动"

2. 让 AI 列出可能的原因（2-3 个假设）
   → 假设 A：事件冒泡未被拦截
   → 假设 B：cardDragRef 未被正确设置
   → 假设 C：SVG 层拦截了鼠标事件

3. 用最小代价验证假设
   → eval 在浏览器中查看 DOM 属性
   → 截图观察视觉状态

4. 确认根因后再实施修复

5. 修复后验证（原场景 + 相关场景）
```

### 3.3 知识空间：结构化的领域上下文

本项目实践表明，**把领域知识结构化存储**是 AI 工具持续有效的关键。知识空间的形式：

```
知识空间/
├── 概念词典.md          # 术语定义：I²C、Hisport、TopoNode、BusinessConnector...
├── 数据模型.md          # 核心数据结构及字段含义
├── 视觉规范.md          # 颜色映射、连线样式、图标含义
├── 已知约束.md          # 业务规则：PSR→IEU 不是直连、Hisport 双信号...
└── 历史决策.md          # 为什么这么设计（不是做了什么，是为什么）
```

**使用方式**：每次新的对话开始时，把相关知识空间文档喂给 AI，让它"进入上下文"再开始工作。

---

### 3.4 风格上下文提取

本项目隐含了一套完整的视觉语言，应当显性化沉淀为**设计 Token 文档**：

```markdown
## 颜色系统
- 背景层：#060a0f（画布）/ #0b1118（卡片）/ #0f1f35（选中卡片）
- 板卡类型色：PSR #7c3aed / BCU #0891b2 / IEU #2563eb / SEU #dc2626 ...
- 节点类型色：connector #16a34a / i2c_mux #7c3aed / eeprom #d97706 ...
- 连线色：I²C虚线 #0ea5e9 / Hisport PCIe #3b82f6 / CEM #f97316 ...

## 卡片结构规范
- 边框：1px solid {typeColor}44（常态）/ {typeColor}（选中）
- 圆角：8px
- 阴影：0 2px 8px #00000066（常态）/ 0 0 16px {typeColor}55（选中）
- 节点图标：22×16px，borderRadius 3px / connector 24×14px borderRadius 7px

## 交互规范
- 拖拽：整个卡片可拖，排除 button/input 等控件
- 选中：单击选中，高亮该卡片及其关联连线
- 悬停连线：crosshair cursor，tooltip 显示连线属性
- 缩放：滚轮，0.15x - 2.5x
```

### 3.5 可封装的组件模式

从本项目提炼出以下可独立复用的组件：

| 组件 | 能力 | 适用场景 |
|---|---|---|
| **InfiniteCanvas** | 平移+缩放+坐标变换，支持拖拽卡片 | 任何需要无限画布的可视化 |
| **ConnectionLineSVG** | bezier 连线，支持 hover/click tooltip，pointerEvents 穿透卡片 | 关系图、拓扑图、流程图 |

| **MiniDiagram** | 在卡片内渲染小型树状/总线图 | 详情卡片的结构预览 |

### 3.6 Demo 开发工作流的核心洞察

#### 洞察 1：原型保真度的选择
AI 可以直接生成**高保真可交互原型**（而不是静态稿），这改变了设计师和工程师之间的沟通方式。设计师可以把可运行的 Demo 直接当作沟通工具，而不是设计稿。

**带来的变化**：
- 从"展示设计稿 → 工程师理解 → 实现" 变为 "展示可运行 Demo → 直接对齐"
- Figma 原型的优先级下降，可运行代码的优先级上升

#### 洞察 2：设计师的新核心能力
在 AI 时代，体验设计师需要建立的新能力不是"会写代码"，而是：

- **问题精确化**：把模糊的体验问题转化为 AI 可以处理的精确描述
- **判断力**：在 AI 给出多个方案时，判断哪个更符合设计意图
- **领域翻译**：把业务/领域知识翻译成 AI 能理解的格式
- **结果验证**：能够识别 AI 的实现是否真正解决了设计问题（而不只是"跑起来了"）

#### 洞察 3：知识积累的复利效应
每次对话结束后沉淀的文档（分析材料文件夹中的 .md 文件）会形成知识资产，下一次工作时喂给 AI 可以大幅减少重新解释的成本。**领域知识积累越多，AI 的输出质量越高，设计师需要校正的次数越少**。

## 四、复杂图标与视觉生成的模式

本项目中大量 SVG 图标和缩略图是通过 AI 生成的（ProjectList 中各视图的 thumb 组件）。提炼出以下模式：

### 图标生成 Prompt 结构
```
场景描述 + 配色约束 + 技术约束（SVG viewBox 尺寸）+ 要传达的语义

示例：
"生成一个 160×100 的 SVG 缩略图，
背景 #080812，表现服务器机箱的等轴测视图，
蓝色调 (#3b82f6/#22d3ee)，
要让人一眼看出这是 3D 可视化入口"
```

### 有效的约束组合
- **语义优先**：先说要传达什么，再说怎么画
- **颜色锚定**：给出具体色值，而不是说"蓝色"
- **尺寸约束**：SVG 的 viewBox 明确，避免比例问题
- **参考已有**：说"风格和其他卡片的缩略图一致"，让 AI 保持一致性

## 六、从大型设计平台部门视角的沉淀机会

> 以上内容是从"设计师个人"视角的提炼。本章从**大企业设计平台部门**的角度思考：这个实验产生了哪些可以被组织层面沉淀、共享和规模化的内容。

### 6.1 可规模化的资产类型

#### 领域知识库（Domain Knowledge Base）
本项目的知识空间文档（概念词典、数据模型、已知约束、历史决策）代表了一类新型资产——**结构化的领域上下文**，专门为 AI 协作而组织。

对设计平台部门的价值：
- 新成员加入时，喂给 AI 即可快速"进入状态"，无需口口相传的知识转移
- 不同产品线可以维护各自的领域知识库，形成组织级知识资产
- 随项目迭代持续更新，形成知识复利

**建议**：建立"知识库模板"——每个新项目启动时，根据模板填写领域知识库的框架结构。

---

#### AI 协作 Skill 库
本项目识别的 Skill-01 到 Skill-05 是通用的提示词模式。对设计平台部门，可以建立：

```
设计平台 AI Skill 库
├── 原型生成类 Skill（数据可视化、交互界面、图标生成）
├── 设计规范类 Skill（从代码库提取 Token、规范一致性检查）
├── 竞品分析类 Skill（截图→结构分析、交互还原）
├── Bug 诊断类 Skill（现象描述→假设生成→验证→修复）
└── 文档生成类 Skill（代码→注释、设计决策→文档）
```

每个 Skill 包含：触发条件、输入格式、提示词模板、预期输出示例、已知局限。

---

#### 可复用组件包（Design Engineering Kit）
从本项目提炼的组件（InfiniteCanvas、ConnectionLineSVG、DraggableCard 等）有潜力成为设计平台的工程资产：

| 组件 | 适用的产品类型 |
|---|---|
| InfiniteCanvas | 任何需要无限画布的可视化工具（拓扑图、流程图、白板） |
| ConnectionLineSVG | 关系图、架构图、流程编排工具 |
| DraggableCard | 看板、节点编辑器、布局工具 |
| MiniBusDiagram | 详情卡片内的结构预览、层次图 |

**建议**：建立"可视化组件 Storybook"，每个组件附带 AI 生成的使用示例和 Prompt 模板（"如何让 AI 帮你扩展这个组件"）。

### 6.2 团队协作模式的升级

#### 从"设计稿评审"到"Demo 评审"
本实验表明，体验设计师可以直接产出**可运行的高保真交互 Demo**，这改变了设计-研发协作的接口形态：

| 传统模式 | AI 辅助模式 |
|---|---|
| 设计师出 Figma 静态稿 | 设计师出可运行 Demo |
| 研发理解设计意图（有损耗） | 研发直接参考 Demo 实现（损耗极小） |
| 反复联调细节 | Demo 即规格，可直接拿代码或对照实现 |
| 设计稿与实现总有偏差 | Demo 本身就是实现，偏差为零 |

**对设计平台部门的建议**：
- 将"AI 辅助 Demo 开发能力"纳入设计师技能矩阵
- 建立"Demo 评审"流程作为"设计评审"的补充或替代
- 培训设计师"如何把设计意图翻译成 AI 可执行的语言"

```
现有流程节点          AI 介入机会
──────────────────────────────────────────
需求收敛              → AI 辅助竞品分析、用户场景梳理

开发移交              → AI 生成标注文档、组件使用说明

```

## 七、3D / 2.5D 场景带来的沉淀机会点

> 本项目的 3D 仿真视图（Three.js 等轴测服务器模型）和 2D 拓扑图的并存，开启了一个新的设计探索维度：**当可视化进入三维空间后，沉淀的方式和机会点有哪些变化？**

### 7.1 3D/2.5D 视图的独特设计价值

#### 为什么硬件工具需要 3D 视图
2D 拓扑图回答的是"板卡之间是什么关系"，而 3D 视图回答的是"这些东西在物理空间中怎么放"：

| 维度 | 2D 拓扑图 | 3D 仿真视图 |
|---|---|---|
| 信息类型 | 逻辑连接关系 | 物理空间关系 |
| 用户心智 | 系统工程师 | 硬件工程师、客户展示 |
| 主要交互 | 点击/筛选/查询 | 旋转/漫游/组件拆解 |
| 核心价值 | 理解架构 | 感知实体、沟通展示 |

**设计机会**：两个视图联动——在 2D 拓扑中点击板卡，3D 视图中对应的物理位置高亮；在 3D 中旋转到某个板卡，2D 拓扑中该板卡的连线展开。

---

### 7.2 3D/2.5D 场景中 AI 协作的特殊机会

#### 机会点 1：等轴测图标与缩略图生成
本项目中 ProjectList 的"3D仿真"入口卡片缩略图（等轴测服务器 SVG）是 AI 生成的，这开启了：

- **等轴测图标系统**：为所有板卡类型生成等轴测风格的图标（如 BCU 板、IEU Riser、SEU 背板），在 3D 视图的 UI overlay 中使用
- **Prompt 模式**：`等轴测视角 + 特定硬件形态 + 颜色约束 + SVG 输出` 是一个可沉淀的 Skill 类型
- **一致性挑战**：不同形态的板卡需要保持同一视角、同一光源、同一风格——这需要在 Prompt 中精确描述"等轴测参数"

**建议沉淀**：建立"等轴测 SVG 图标 Prompt 模板"，包含视角参数（30° 倾角、120° 方位角）、光源方向、阴影参数、色调一致性要求。

---

#### 机会点 2：3D 场景中的信息层级设计
3D 视图的信息密度控制比 2D 更难——物体在空间中遮挡彼此，视角变化时信息可读性变化剧烈。AI 可以辅助：

- **LOD（Level of Detail）设计**：近处显示详细信息（型号、状态灯），远处简化为颜色块——AI 可以帮助生成不同 LOD 级别的组件
- **透明度策略**：选中某板卡时，前方遮挡物自动透明化（X-Ray 模式）——AI 可以生成 Three.js 材质切换的实现代码
- **标注浮层**：3D 中的 billboard 标签（始终面向相机）+ 引导线——AI 生成自适应相机角度的 label 组件

---

#### 机会点 3：2D→3D 的数据模型复用与视角对齐
本项目已有结构化的 `HwBoard` 数据模型（含板卡类型、连接关系）。这套数据可以驱动 3D 场景：

```
HwBoard 数据 → 3D 场景生成规则
─────────────────────────────────
board.type     → 3D 模型选择（BCU 模型、IEU 模型...）
board.uid      → 3D 对象命名（可双向联动）
BOARD_CONNECTIONS → 3D 中的连线/光纤可视化
soft.eventCount → 状态灯动画（正常=绿、告警=黄）
```

**AI 辅助机会**：描述"根据 `BOARD_CONNECTIONS` 数据，生成 Three.js 中板卡之间的光纤连线动画"——AI 可以直接利用已有数据结构生成 3D 可视化逻辑。

**建议沉淀**：整理一份"数据驱动 3D 场景"的映射文档，定义 HwBoard 各字段到 Three.js 场景属性的对应关系。

---

#### 机会点 4：3D 视图的交互模式 Prompt
3D 交互（旋转、平移、缩放、拆解动画、碰撞检测）的实现细节多、调参复杂，是 AI 辅助的高价值区：

```
可沉淀的 3D 交互 Skill：
├── OrbitControls 参数调优（惯性、阻尼、角度限制）
├── 对象选中 + Raycasting（点击 3D 对象触发 2D 联动）
├── 爆炸图动画（Exploded View，按轴向展开各层板卡）
├── 相机路径动画（Camera Path，漫游展示整机）
└── 软阴影与环境光调优（让硬件模型看起来"真实"）
```

每个 Skill 包含：Three.js 代码框架 + 关键参数的调节建议 + 常见错误的诊断清单。

---

### 7.3 等轴测（2.5D）作为中间层的设计机会

等轴测视图介于 2D 和 3D 之间，有独特的设计优势：
- 无透视变形，信息密度高，可读性优于真 3D
- 可以用纯 SVG/Canvas 实现，不需要 WebGL（性能更好，可嵌入任何网页）
- 技术栈门槛低，AI 辅助生成的成功率更高

**本项目中的等轴测元素**：ProjectList 的 ThreeDThumb 缩略图（等轴测服务器）是纯 SVG，AI 一次生成。

**可以沉淀的等轴测 Skill**：
```
输入：板卡或机箱的几何描述（长宽高、面板元素位置）
输出：等轴测 SVG（30° 视角，三面可见，含阴影）
用途：硬件文档配图、UI 中的产品缩略图、汇报 PPT 的架构示意图
```

**建议探索**：能否从 HwBoard 数据生成整机等轴测示意图？以板卡类型为层，按物理位置（slot 信息）排列，AI 生成等轴测 SVG。

---

### 7.4 3D/2.5D 场景对知识库的特殊要求

3D 场景需要额外的知识空间内容：

```
知识空间（3D 扩展）
├── 物理尺寸约束.md    # 板卡的实际物理尺寸（U、半高/全高）
├── 槽位布局.md        # 机箱中各板卡的物理位置关系（slot 编号）
├── 3D 资产清单.md     # 已有的 .glb 模型文件、对应的板卡类型、精度级别
└── 光照与材质规范.md  # 金属/塑料/PCB 材质参数，光源设置（HDR 还是点光源）
```

**关键约束**：3D 场景的"正确性"标准不同于 2D——2D 拓扑错了是逻辑错误（工程师可以发现），3D 视图错了可能是外观错误（可能误导硬件工程师对物理安装的判断）。**3D 知识库需要包含物理正确性验证的方法**。

## 八、核心结论

这个实验最重要的结论是：

> **体验设计师 + AI 工具的最佳协作模式，不是"设计师学会用 AI 写代码"，而是"设计师用 AI 把设计判断力直接变成可运行产物"。**

设计判断力（为什么这样做）不能被 AI 替代，但实现能力（怎么做到）可以被 AI 极大地加速。设计师的价值在于持续提供正确的判断——对领域语义的判断、对视觉体验的判断、对用户需求的判断——而 AI 把这些判断快速物化。

这要求设计师发展出新的核心技能：**精确描述问题、验证 AI 输出、积累领域知识**。这三件事做好了，设计师可以独立完成过去需要一个设计+开发团队才能完成的工作。

从设计平台部门的视角，这个实验的更大意义在于：**AI 协作能力是可以被组织化、规模化的**。知识库、Skill 库、组件库、SOP——每一层都可以从个人实践上升为团队资产，从一个项目的经验上升为跨产品线的基础设施。
