## 1. SMC 偏移量计算器

### 1.1 背景：什么是 BMC 和 SMC 命令字？

#### BMC 是服务器的"隐形管理员"

服务器里有一块专用芯片叫做 **BMC（Baseboard Management Controller，底板管理控制器）**。它和主 CPU 完全独立，即使服务器断电也在运行，承担着"监控和控制整台服务器硬件"的职责，相当于服务器的"神经中枢"。

BMC 日常工作包括：读取 CPU 温度、控制风扇转速、检测内存是否报错、记录硬件故障日志…… 它需要和数十甚至数百个硬件组件实时通信，包括温度传感器、电源模块、风扇控制器、存储背板等。

#### SMC 命令字在整个技术栈里的位置

BMC 固件并不直接操作 I2C 寄存器，而是通过多个抽象层来访问硬件：

```
┌──────────────────────────────────────────┐
│ 固件应用层：SMC 命令字 (32-bit offset)    │ ← 这个工具处理的层
│   "我要读哪个 CSR 对象的哪个属性"          │
└────────────────┬─────────────────────────┘
                 │ 中间件将命令字解析，定位 CSR 对象
                 ↓
┌──────────────────────────────────────────┐
│ CSR 对象层：拓扑视图展示的逻辑关系         │ ← 拓扑图处理的层
│   Fan_1 / Scanner_CpuTemp / Accessor_…  │
└────────────────┬─────────────────────────┘
                 │ CSR 对象绑定了真实的硬件访问方式
                 ↓
┌──────────────────────────────────────────┐
│ 硬件驱动层：I2C / PECI / IPMI 等物理协议  │ ← 物理通信层
│   哪条总线、哪个芯片地址、哪个寄存器        │
└────────────────┬─────────────────────────┘
                 ↓
             传感器 / 风扇 / 电源模块 等物理硬件
```

**I2C 拓扑**（CSR Topology 视图）描述的是软件逻辑层：哪些 CSR 对象存在、它们之间如何引用。I2C 是底层物理协议，对应用层透明。

**SMC 命令字发送的对象**不是 I2C 芯片，而是 **CSR 对象**（如 `Fan_1`、`Scanner_CpuTemp`）——这些是 BMC 固件对硬件的软件抽象。命令字是访问"某个 CSR 对象的某个属性"的唯一索引键。

#### "按位打包"是什么意思？

当 BMC 要和某个 CSR 对象交互时，需要一套精确的二进制协议：**每次访问都需要发送一个 32-bit 整数作为"访问地址"，这个整数就叫 SMC 命令字（偏移量 Offset）**。

可以把这个 32-bit 整数理解为一张"门牌"——门牌上不同的位段分别编码了"找哪个部门"、"部门里的哪个操作"、"是查询还是下令"、"带什么附加参数"。BMC 中间件拿到这个门牌，就能精确定位应该访问哪个 CSR 对象的哪个属性。

#### 先读懂数字格式：32-bit 整数、十六进制、0xC 里的 C 分别是什么

在看字段结构之前，需要先搞清楚几个数字表示的基础概念——即使有技术背景的人，这里也容易混淆。

**计算机只认识二进制（0 和 1）**

计算机内部用**二进制**（只有 0/1）存储一切数据。"32-bit 整数"就是用 **32 个二进制位（bit）** 表示的整数。32 个 bit 能表示的最大值是 2³² − 1 = 4,294,967,295（约 43 亿）。

例如，数字 12 用 8 个二进制位写出来是：

```
0  0  0  0  1  1  0  0
↑                    ↑
高位(值小)          低位(值大)
```

32 个 0/1 排在一起，人类几乎不可能直接阅读——所以工程师用**十六进制**作为简写系统。

---

**十六进制的核心规则：每 4 个二进制位 = 1 个字符**

4 个二进制位恰好有 2⁴ = **16 种**组合（0000 到 1111），十六进制为这 16 种状态准备了 16 个字符。数字 0~9 只够用 10 个，剩下 6 个状态（10~15）借用了字母 **A、B、C、D、E、F**：

| 十六进制字符 | 十进制值 | 二进制 |
|:-----------:|:--------:|:------:|
| 0 | 0  | 0000 |
| 1 | 1  | 0001 |
| 2 | 2  | 0010 |
| 3 | 3  | 0011 |
| 4 | 4  | 0100 |
| 5 | 5  | 0101 |
| 6 | 6  | 0110 |
| 7 | 7  | 0111 |
| 8 | 8  | 1000 |
| 9 | 9  | 1001 |
| A | 10 | 1010 |
| B | 11 | 1011 |
| **C** | **12** | **1100** |
| D | 13 | 1101 |
| E | 14 | 1110 |
| F | 15 | 1111 |

**`0xC` 的 C 就是十进制的 12，二进制的 1100。**

`A–F` 本身没有特别含义，只是沿用了字母表前 6 个字母，属于工程领域的历史约定。`0x` 是前缀标记（源于 C 语言），告诉读者和编译器"后面的数字是十六进制"——在嵌入式固件、芯片手册、协议规范里随处可见。

所以看到 `0x0C`，拆开读就是：十六进制的 `0`（=0）和 `C`（=12），合在一起是一个 8-bit（两个十六进制字符）的数值，等于十进制的 12。

---

**32-bit 整数 = 固定 8 个十六进制字符**

每 4 个 bit = 1 个十六进制字符，所以：

> 32 位 ÷ 4 = **8 个十六进制字符**（固定长度，不足则左边补 `0`）

```
32位二进制（每4位切一组）：
  0011 │ 0000 │ 0100 │ 0100 │ 0000 │ 0001 │ 0000 │ 0000
    ↓       ↓       ↓       ↓       ↓       ↓       ↓       ↓
    3       0       4       4       0       1       0       0

合并加前缀 → 十六进制：0x30440100
同一个数  → 十进制：  809,893,888
同一个数  → 二进制：  00110000010001000000000100000000（32位）
```

三种写法代表**完全相同的数值**，只是给不同场景的人看：

| 格式 | 示例 | 适用场景 |
|------|------|---------|
| 十进制 | `809893888` | 日志系统、数据库（默认格式） |
| 十六进制 | `0x30440100` | 固件源码、协议文档、芯片手册 |
| 二进制 | `00110000…00` | 机器存储的真实形态，人类直接看太费力 |

十六进制在硬件领域成为标准的原因：长度固定（32-bit 永远是 8 字符），且可以在脑中直接和二进制互换（`0xF = 1111`，`0xC = 1100`），而十进制做不到这一点。

---

#### 32-bit 命令字的内部结构——5 个字段各司其职

**什么是"位编号 [31:26]"？**

一个 32-bit 整数的 32 个 bit，硬件工程师从右到左依次编号（最右边的 bit 最"轻"，是 bit 0；最左边的 bit 最"重"，是 bit 31）：

```
位编号：  31  30  29  28  27  26  …  9   8   7   6   5   4   3   2   1   0
权重：   2³¹ 2³⁰ 2²⁹ 2²⁸ …                              …  2³  2²  2¹  2⁰
方向：   ←────────── 最高位 (MSB，Most Significant Bit)    最低位 (LSB) ──────→
```

记法 `[31:26]` 表示"第 31 位到第 26 位，共 6 个 bit"——这是芯片数据手册和 Verilog/VHDL 硬件描述语言的通用写法。工程师说"把功能码放在 [31:26]"，意思是把功能码的数值**左移 26 位**，让它占据这段位置：`fn << 26`。

这 32 位被**无重叠、无空隙**地切割成 5 个字段（6+16+1+1+8 = 32）：

```
位编号  [31:26]     [25:10]         [9]        [8]        [7:0]
字段    功能码       命令码          读取方式    读写方向    参数
宽度    6 位         16 位          1 位        1 位        8 位
范围    0x00–0x3F   0x0000–0xFFFF  0 或 1      0 或 1      0x00–0xFF
```

**字段之间完全不重叠**：功能码占 [31:26]，命令码紧接在 [25:10]，没有任何 bit 被两个字段共用。

**功能码 Function（6-bit，0~63）——"找哪个部门"**

代表这条命令属于哪个硬件子系统。比如功能码 `0x01` 可能对应"温度管理模块"，`0x02` 对应"风扇控制模块"，`0x03` 对应"电源管理模块"。一块服务器主板上有几十种功能模块，功能码 6 位最多能寻址 64 个部门。

**命令码 Command（16-bit，0~65535）——"找部门里的哪个操作"**

在某个功能模块内，有许多不同的具体操作。比如在"温度管理"这个部门里，命令码 `0x000A` 可能是"获取 CPU 核心温度"，`0x000B` 是"获取内存温度"，`0x000C` 是"设置过温保护阈值"。16 位意味着一个模块最多支持 65536 个不同命令，足以覆盖最复杂的硬件。

**读取方式 MS（1-bit）——"单读还是广播读"**

`0` = 多读（Multi-read）：一次请求读取多个组件的同一类数据，效率更高。
`1` = 单读（Single-read）：只读取一个具体组件的数据，精确但较慢。

类比：是发一封群邮件问所有部门的月报（多读），还是分别给每个部门发一封（单读）。

**读写方向 RW（1-bit）——"查询还是下命令"**

`0` = 写（Write）：向硬件下发指令，改变其状态（如"把风扇转速调到 80%"）。
`1` = 读（Read）：向硬件查询当前状态（如"现在风扇转速是多少"）。

**参数 Parameter（8-bit，0~255）——"附带的具体数值"**

一些命令需要携带一个附加参数。比如"设置风扇转速"这个写操作，参数就是目标转速百分比（0~255 映射到 0~100%）；或者指定读取的是第几号传感器（0~255 个）。

#### 十六进制字符的边界 ≠ 字段边界（这是关键！）

上面已经解释了十六进制的基础（每字符 = 4 bit，32-bit = 8 字符）。但有一个非常重要的现实：**十六进制字符的切割位置和命令字字段的边界并不对齐。**

以 `0x30440100` 为例，把 8 个十六进制字符和字段对照排列：

```
十六进制：     3      0      4      4      0      1      0      0
             ↓      ↓      ↓      ↓      ↓      ↓      ↓      ↓
二进制：    0011   0000   0100   0100   0000   0001   0000   0000
位编号：   31-28  27-24  23-20  19-16  15-12  11-8   7-4    3-0

字段归属：
  '3' (31-28)：全部 → fn
  '0' (27-24)：bits 27-26 → fn 的末尾 2 位
               bits 25-24 → cmd 的开头 2 位   ← 一个字符跨越两个字段
  '4' (23-20)：全部 → cmd
  '4' (19-16)：全部 → cmd
  '0' (15-12)：全部 → cmd
  '1' (11-8) ：bits 11-10 → cmd 的末尾 2 位
               bit  9     → ms
               bit  8     → rw                ← 一个字符同时包含三个字段
  '0'  (7-4) ：全部 → param
  '0'  (3-0) ：全部 → param
```

**字段本身不重叠**（fn 用完 [31:26]，cmd 紧接着从 [25:10]，互不侵犯），但**十六进制字符的切割方式和字段边界不一致**——第 2 个字符 `'0'` 内部同时包含 fn 的尾部和 cmd 的头部；第 6 个字符 `'1'` 内部同时包含 cmd 的尾部、ms 和 rw。

这意味着：**工程师不能直接从十六进制字面读出各字段的值，必须做位运算（移位 + 掩码）。** 这正是这个工具存在的核心价值——把不可见的字段边界变成可见的独立输入框。

**为什么日志里又出现十进制？**

写代码时工程师用十六进制（与协议文档对齐），但运行时产生的日志和监控数据往往是十进制——因为多数日志框架、数据库、运维看板默认用整数类型存储和展示数字。同一个值，在不同系统里以不同进制出现：

| 出现场景 | 格式 | 示例 |
|---------|------|------|
| 固件源码 / 协议文档 | 十六进制 | `0x30440100` |
| 系统日志（syslog、内核日志） | **十进制** | `offset=809893888` |
| 故障数据库记录 | **十进制** | SQL `int` 类型默认十进制显示 |
| 运维告警工单 | **十进制** | 非技术人员填写 |

工程师在不同阶段会拿到同一个值的不同表示。工具同时展示两种格式，消除"看到十进制数需要先打开计算器换算"的流程中断。

### 1.2 具体使用场景

#### 场景一：开发阶段——"我要访问某个传感器，该填什么偏移量？"

工程师拿到协议文档，知道"读取 CPU 入口温度"的字段值：
- 功能码 = `0x0C`（温度管理）
- 命令码 = `0x0060`（入口温度命令）
- 读取方式 = `0`（单读）
- 读写方向 = `1`（读操作）
- 参数 = `0x01`（第1号CPU）

但代码里需要的是一个 32-bit 整数。工程师必须手动计算：

```
(0x0C << 26) | (0x0060 << 10) | (0 << 9) | (1 << 8) | 0x01
= 0x03000000 | 0x00018000 | 0x000 | 0x100 | 0x01
= 0x03019101   ← 这个数字写进源代码
```

移位、OR 操作一步都不能出错。

#### 场景二：调试阶段——"日志里出现了一个数字，它代表什么操作？"

线上服务器报错，日志显示某条 SMC 命令失败，偏移量是 `809893888`（十进制）。工程师需要快速判断是哪个功能出了问题，要能解析出各字段。手工解析过程：

```
809893888 → 0x30440100（十六进制）
→ 右移26位 = 功能码 0x0C
→ 右移10位 & 0xFFFF = 命令码 0x0011
→ 第9位 = MS 0
→ 第8位 = RW 1（读操作）
→ 低8位 = 参数 0x00
```

没有计算器，很少有人能在脑子里做完这些位运算。

#### 场景三：代码 Review——"这个魔数是什么意思？"

同事代码里写了：
```c
smc_read(0x30280001);
```

Reviewer 看到 `0x30280001` 这个"魔数"，不查文档完全不知道它的含义，更无法判断字段值是否合理。

| 场景 | 操作方向 | 触发 | 期望输出 |
|------|----------|------|----------|
| 开发 | 编码（字段 → 偏移量） | 已知各字段，需要 32-bit 整数 | `0x30280001`（写入代码） |
| 调试 | 解码（偏移量 → 字段） | 日志中出现十进制数 | 各字段含义，快速定位问题模块 |
| Review | 解码 | 代码中出现十六进制魔数 | 字段含义，验证合理性 |

### 1.3 目标用户

| 用户类型 | 使用频率 | 核心诉求 | 典型触发点 |
|----------|----------|----------|-----------|
| **BMC 固件工程师** | 高频，开发期每天多次 | 精确、快速，不想手算 | 新写一个硬件访问接口 |
| **硬件调试工程师** | 中频，调试期集中 | 快速反向解析日志数字 | 故障排查时看到神秘数字 |
| **代码 Review 者** | 低频 | 快速理解魔数语义 | 审查其他人的代码 |

**用户共同特征**：有较强的技术背景，对计算准确性要求极高，对 UI 审美要求不高，但对**效率和零错误**有强迫症般的要求。

### 1.4 用户痛点分析

#### 痛点①：手工移位计算极易出错，且错了很难发现

32-bit 的移位计算涉及多步操作。以下是工程师真实的心算过程：

```
功能码 0x0C → 0x0C << 26 = ?
0x0C = 12 (十进制)
12 * 2^26 = 12 * 67108864 = 805306368
805306368 = 0x30000000 ✓

命令码 0x0060 → 0x0060 << 10 = ?
0x60 = 96, 96 * 1024 = 98304
98304 = 0x18000 ✓

最终 OR 合并... (继续...)
```

一旦某步算错（如忘记某个字段的移位量），最终数字看起来仍然是"合法的 32-bit 整数"——编译器不报错，静态分析工具不报错，只有运行时访问错误硬件地址时才会爆出 Bug。而定位这个 Bug 往往要花数小时。

**痛点核心：错误完全隐蔽，代价极高。**

#### 痛点②：日志中的十进制数字对人脑是"不透明"的

十六进制 `0x30440100` 一眼就能看出功能码（前两位是 `30`），但十进制 `809893888` 毫无规律可言。工程师被迫在脑内执行"十进制 → 十六进制 → 拆字段"的三步转换，不借助工具几乎不可能快速完成。

现实情况是：很多工程师会打开 Python，输入：
```python
>>> hex(809893888)
'0x30440100'
>>> (0x30440100 >> 26) & 0x3F  # 功能码
12
>>> (0x30440100 >> 10) & 0xFFFF  # 命令码
...
```

**痛点核心：每次调试都需要临时搭脚手架，中断工作流。**

#### 痛点③：字段含义对非原始开发者不透明

命令字由原始开发者设计，但代码在整个团队中流转。新加入的工程师、跨团队的 Reviewer 看到 `0x30280001`，完全不知道这代表什么功能。询问原作者需要时间，查文档需要知道去哪查——这产生了大量隐形的沟通成本。

**痛点核心：知识孤岛，代码可读性差，新人上手成本高。**

#### 痛点④：当前工具只在 VSCode 扩展内可用

原版工具是 VSCode Webview 插件，只能在安装了该扩展的机器上使用。当工程师用其他编辑器、在 Code Review 平台查看、或在另一台测试机上调试时，工具完全不可用——退回到纸和 Python 的时代。

### 1.5 设计机会点

#### 🔴 高优先级

**A. 缺少位图可视化**
当前仅有一行文字说明格式。用户输入字段时，无法直观感知"我填的这个值会影响 32 位中的哪些 bit"。尤其对于初次接触的人，字段与整数之间的关系完全是黑盒。
> **建议**：在结果区上方增加 32-bit 位图（类似芯片手册寄存器图）：8 个十六进制格子横排，不同颜色区分各字段范围；用户修改任一字段时，对应颜色块实时高亮变化，让"位"变得可见。

**B. 解码入口可发现性低、标题有歧义**
"10进制偏移量 → 解析到组成字段"是调试场景的核心需求，但在 UI 上视觉权重低于编码区。且标题"10进制偏移量"不准确——实际上也支持十六进制输入（工具内部自动识别 `0x` 前缀）。日志里经常同时出现两种格式，用户不应被这个限制性标题误导。
> **建议**：将解码区升级为与编码区并列的一级区块，改名为"反查偏移量"；输入框 placeholder 改为"支持十进制或 0x 十六进制"。

#### 🟡 中优先级

**C. 无历史记录**
调试时会反复输入不同参数组合对比，每次关闭或刷新都归零。
> **建议**：用 localStorage 存储最近 10 条输入（字段值 + 结果），底部显示历史列表，点击一键回填。

**D. 结果复制格式不可定制**
当前固定输出 `0x30280001 (807141377)`，但写代码时只需要 hex，记录文档时可能只需要 dec。
> **建议**：复制按钮旁加下拉，提供"仅 Hex / 仅 Dec / 两者"三种格式选项。

**E. 无功能码 / 命令码语义参考**
工程师输入功能码 `0x0C` 时，工具本身无法告知"0x0C 代表温度管理模块"——这需要工程师自己查文档。
> **建议**：支持导入一份 JSON 速查表（由团队维护），使得鼠标悬停功能码字段时，显示该值对应的模块名称。

#### 🟢 低优先级

**F. 字段布局不够直观**
命令码（16-bit）比其他字段宽，当前处理方式是让它 `span 2` 列，但与其他字段的视觉节奏不匹配。
> **建议**：将布局改为"功能码 + 命令码"一行（突出核心字段），"MS + RW + 参数"另一行（辅助字段），保持视觉对称。

## 2. 批量表达式计算器

### 2.1 背景：什么是管道表达式？

#### CSR 配置中的"数据处理流水线"

在 BMC 固件配置体系（CSR，Configuration & Status Register）中，有大量的数据需要在"传感器原始值"和"最终展示/使用值"之间做转换处理。

举一个具体例子：某温度传感器返回的原始值是 `75`（一个整数），但最终这个值需要：
1. 加上校准偏移量（比如 +5）
2. 转成十六进制格式（因为下游系统用十六进制存储）
3. 再转成大写（协议规定）

这三步操作，如果用传统编程语言，需要写一段代码。但 CSR 配置里用的是一种更紧凑的**管道表达式（Pipe Expression）**语法——借鉴了 Unix 命令行的 `|` 管道思想：

```
$1 | add 5 | toHex | string.upper
```

这一行表达式的含义是：
- `$1`：取第一个输入值（即原始传感器值，例如 `75`）
- `| add 5`：加上 5，得到 `80`
- `| toHex`：转成十六进制，得到 `"0x50"`
- `| string.upper`：转成大写，得到 `"0X50"`

**每一个 `|` 之后的操作，都以上一步的输出作为输入**，形成一条从左到右的数据流水线。

#### 为什么叫"批量"表达式计算器？

"批量"有两层含义：

1. **表达式可以有多个输入参数（$1、$2、$3...）**：一个表达式可能同时依赖多个传感器的值。比如计算两个温度的均值，需要 $1（CPU温度）和 $2（环境温度）同时输入。

2. **验证阶段需要批量跑测试用例**：一个表达式写好后，需要用几十甚至上百组"已知输入 + 已知预期输出"来验证它是否正确。这些测试用例需要批量执行，不能一条条手动对比。

### 2.2 两种工作模式的含义

#### 调试模式（Debug Mode）——"我写了一个表达式，它是怎么一步步计算的？"

调试模式面向**开发阶段**。工程师正在编写一个新的管道表达式，不确定每一步操作是否符合预期，需要看清楚中间过程。

**工作方式**：
1. 在表达式输入框里填写表达式，如 `$1 | add $2 | toHex 8 | string.upper`
2. 在下方填写各输入参数的当前值：`$1 = 100`，`$2 = 50`
3. 工具实时显示每一个阶段的中间结果：

```
阶段 1：$1           → 100        （取输入值）
阶段 2：add $2       → 150        （加上 $2=50）
阶段 3：toHex 8      → "0x00000096" （转8位十六进制）
阶段 4：string.upper → "0X00000096" （转大写）
```

如果某步出错，错误会精确标注在哪个阶段，而不是只告诉你"表达式失败了"。

**类比**：就像代码调试器的"单步执行"——不是直接看最终结果，而是一步步观察程序的行为。

#### 用例模式（Test Case Mode）——"我的表达式对所有输入都正确吗？"

用例模式面向**验证阶段**。表达式已经写好，工程师需要用大量真实数据验证它的正确性，相当于运行单元测试。

**工作方式**：
1. 表达式保持不变（复用调试模式里写好的）
2. 在文本框里粘贴一批测试用例，格式为：每行一个用例，最后一列是预期输出：
   ```
   100 50 0X00000096
   200 30 0X000000E6
   0   0  0X00000000
   ```
   第一行含义：输入 $1=100，$2=50，预期结果是 `0X00000096`
3. 点击"批量执行"，工具对每一行输入运行表达式，对比实际结果与预期输出，输出结果表：

| # | 输入 | 预期 | 实际 | 状态 |
|---|------|------|------|------|
| 1 | 100 50 | 0X00000096 | 0X00000096 | ✅ |
| 2 | 200 30 | 0X000000E6 | 0X000000E6 | ✅ |
| 3 | 0 0 | 0X00000000 | 0X00000000 | ✅ |

**类比**：就像单元测试框架（JUnit、pytest）——把已知的输入输出对批量跑一遍，看哪条失败了。

#### 输入参数（$1、$2…）的作用是什么？

管道表达式里的 `$1`、`$2` 是**占位变量**，代表"在运行时动态传入的值"。这些值来自外部（传感器读数、配置参数、另一个 CSR 对象的值等）。

工具里的"输入参数"区域就是让你模拟这些运行时值：填入 `$1 = 75`，就等于告诉工具"假设现在传感器读数是 75，看看表达式会输出什么"。

#### 为什么要添加多个输入参数？

一条表达式可以引用多个外部数据源。例如：

```
$1 | add $2 | div 2 | toHex
```

这条表达式计算两个传感器值的平均值并转为十六进制：
- `$1` = CPU 核心温度（来自某个传感器）
- `$2` = 环境温度（来自另一个传感器）
- 两者相加除以 2 取均值，再转成十六进制

工具界面中的"+ 添加"按钮，是为了支持这种**多输入**场景——每点一次添加一个新的输入槽（`$2`、`$3`……），工程师可以为每个输入槽填入独立的测试值。

单个参数的场景最常见；两个参数用于"比较"或"合并"场景；三个或以上参数较少但确实存在（如加权平均三路温度）。

### 2.3 具体使用场景

#### 场景一：开发一个新表达式（调试模式的核心场景）

工程师需要为新硬件写一个温度转换表达式。协议要求：原始 ADC 采样值需要减去 40（基准偏移），再乘以 0.125（分辨率系数），再格式化为"%.1f°C"的字符串。

他在调试模式下尝试：
```
$1 | sub 40 | mul 0.125 | string.format "%.1f°C"
```

输入 `$1 = 375`（一个真实的 ADC 采样值），看到各阶段输出：
```
$1            → 375
| sub 40      → 335
| mul 0.125   → 41.875
| string.format "%.1f°C" → "41.9°C"  ← 结果是否符合预期？
```

如果结果不对，他能立即看到是哪一步出了问题，而不是在整条表达式上盲目猜测。

#### 场景二：修改了表达式，需要保证没有破坏原有行为（用例模式的核心场景）

工程师对一个已有表达式做了优化（改变了计算顺序），担心影响边界情况的处理。他从历史测试记录里找到 50 组"输入→预期输出"对，粘贴进用例模式，一键执行，看到"50/50 通过"或"第 37 条失败"——立即知道修改是否安全。

#### 场景三：调试线上异常值（调试模式 + 多参数）

线上某台服务器报告了一个不合理的风速计算值。工程师怀疑是表达式的某个阶段出了问题，把怀疑的表达式粘贴进调试模式，填入出问题时的实际传感器读数，逐阶段追踪哪一步产生了异常输出。

| 模式 | 适用阶段 | 工程师的问题 | 工具的回答 |
|------|----------|-------------|-----------|
| 调试模式 | 开发/Bug复现 | 表达式的第几步出了问题？ | 每步的中间结果 |
| 用例模式 | 测试/回归验证 | 表达式对所有情况都正确吗？ | ✅/❌ 结果表 + 通过率 |

### 2.4 目标用户

| 用户类型 | 频率 | 核心诉求 | 典型场景 |
|----------|------|----------|---------|
| **BMC 固件工程师** | 高频，开发期每天 | 即时反馈、中间值可见 | 编写新的数据转换表达式 |
| **测试/验证工程师** | 中频，测试迭代期 | 批量运行、失败快速定位 | 验证表达式改动没有回归 |
| **系统架构师** | 低频，设计评审 | 快速验证复杂表达式可行性 | 评审方案时当场验证逻辑 |

### 2.5 用户痛点分析

#### 痛点①：没有工具时，开发表达式是"盲飞"

在没有这个工具之前，工程师只能把表达式写入固件代码，编译（通常需要几分钟），部署到测试机，触发对应场景，看日志……才能知道表达式是否正确。一次迭代最短也要 5~10 分钟。一个稍复杂的表达式，往往要迭代 5~10 次才能调对——意味着一个小时就这么没了。

**痛点核心：开发反馈周期极长，改一行配置等同于一次完整的编译+部署。**

#### 痛点②：管道的中间状态完全不可见，出了错不知道怪谁

管道表达式一旦有错，固件日志通常只报告"最终结果异常"，不会告诉你是第 2 步还是第 4 步出了问题。工程师被迫逐段注释、逐段测试，相当于用排除法在黑盒里摸索。

**痛点核心：表达式对工程师是黑盒，调试需要大量"人肉二分法"。**

#### 痛点③：批量验证完全依赖人工

有时候一个表达式需要覆盖 20~50 组边界情况（温度 0°C / 最大值 / 负数 / 边界 ADC 值…）。没有工具时，工程师要么写临时脚本（本身就是额外工作），要么逐条手动验证（容易遗漏，且无法追溯）。

**痛点核心：验证过程非标准化，覆盖率无法保证，且不可复用。**

#### 痛点④：测试用例无处留存

今天写了 30 条测试用例验证了一个表达式，下次修改时这 30 条用例早已丢失——没有地方存放它们。每次都要重新想"应该测哪些边界"。

**痛点核心：测试知识无法积累，每次验证都是从零开始。**

#### 痛点⑤：多输入参数的组合测试手工做几乎不可能

当表达式有 $1 和 $2 两个输入时，要覆盖所有组合（低-低、低-高、高-低、高-高、边界值……），手工测试的工作量呈指数增长。

**痛点核心：多参数场景下，全面验证的成本与参数数量成指数关系。**

### 2.6 设计机会点

#### 🔴 高优先级

**A. 调试模式与用例模式工作流割裂**
这是两个模式切换时的最大断点：调试模式调好的表达式，切换到用例模式后表达式还在，但调试模式的中间结果消失了。更关键的是，用例模式里某条用例跑失败了，工程师想"打开调试看看这组输入的中间状态"，必须手动复制输入值、切换模式、重新粘贴——三次操作打断了排查流程。
> **建议**：不再是 Tab 切换，改为上下区域：上方始终是调试面板（表达式 + 中间结果），下方是用例批量区。用例区某行失败时，提供"在调试区打开"按钮，一键将该行输入填充到调试区并自动运行。

**B. 测试用例输入格式脆弱**
当前格式"空格分隔，最后一列为预期输出"，当 $1=100，$2=50，预期="0X00000096" 时，文本为 `100 50 0X00000096`——新手很容易不知道哪列是预期、哪列是输入；多个参数时更容易数错列。
> **建议**：改为带表头的表格式输入（每列一个输入 + 最后一列预期），或支持 CSV 粘贴（逗号分隔，带错误定位到"第 N 行第 M 列"）。

**C. 失败结果无字符级 diff**
`expected: "0X00000096"` vs `actual: "0x00000096"` ——差了一个大小写，但整行用红色背景标注，工程师还是要靠肉眼找这个区别。
> **建议**：对 expected 与 actual 字符串做字符级 diff，用红/绿底色标出差异字符。

#### 🟡 中优先级

**D. 函数速查可发现性差**
所有支持的函数（add / toHex / string.format …）被压缩在一行小字提示里，没有说明每个函数的签名（参数个数、类型、示例）。工程师记不住不常用的函数，但也不知道去哪查。
> **建议**：增加"函数参考"侧边栏，以分组卡片展示所有函数（算术 / 转换 / 字符串 / 自定义），每个函数含签名 + 示例，点击可将示例插入到表达式光标处。

**E. 无表达式历史**
调试过程中的中间版本会永久丢失，无法回溯到"之前那个版本"。
> **建议**：localStorage 自动保存最近 20 条表达式（含时间戳），工具栏提供历史下拉，支持恢复。

**F. 多输入变量管理笨拙**
"添加/移除"按钮只能操作最后一个输入，无法为输入变量命名（让 $1 被一个有意义的名字代替），无法一眼看出各 $N 分别代表什么物理量。
> **建议**：允许为每个 $N 附加一个"备注标签"（如 `$1 [CPU温度]`），在管道阶段展示时同步显示标签，提升可读性。

#### 🟢 低优先级

**G. 管道阶段连接视觉不够强**
各阶段用左侧色条区分，但没有箭头连线，"管道流"的视觉隐喻不够强烈，用户需要自己理解"上一步的输出是下一步的输入"。
> **建议**：相邻阶段之间加 `→` 符号或微动画过渡，强化流的概念。

**H. 导出格式单一**
"复制结果"只输出 TSV，但有的场景需要 Markdown 表格（贴文档）或 JSON（存数据库）。
> **建议**：导出格式提供下拉：TSV / Markdown 表格 / JSON 数组。

## 3. 能效调速配置模板

### 3.1 背景：服务器风扇调速是个多复杂的问题？

#### 风扇不是开关，而是一套精密的控制系统

服务器里有多组风扇，负责带走 CPU、内存、硬盘产生的热量。风扇不能一直全速转（噪音大、能耗高、寿命短），也不能转太慢（设备过热宕机）。BMC 需要实时根据各种温度传感器的读数，动态决定每组风扇应该转多快——这就是"调速"。

但"调速"听起来简单，实际上是一套相当复杂的控制逻辑：

- **不同位置的温度传感器**监控不同设备（CPU 进风口温度、CPU 核心温度、硬盘温度、内存温度……）
- **不同工作模式**下调速目标不同（节能模式容忍更高温度、高性能模式要求更低温度）
- **多组风扇**可能分布在机箱不同区域，每组负责不同设备的散热
- **异常处理**：如果某个风扇坏掉，其他风扇需要自动拉高转速补偿
- **PID 控制曲线**：温度和转速之间不是简单线性关系，需要一条精确的折线曲线

#### 所有这些逻辑都用一个 YAML 文件描述

BMC 的调速配置存在一个结构化的 YAML 文件里，涉及 8 类实体：

```yaml
slot_id: 1                    # 这台机器的槽位编号
cooling_config:               # 全局调速开关和限制参数
  smart_cooling_state: ...
  
cooling_requirements:         # 温度监控点：每个传感器一条，说明监控什么温度
  - id: 1
    temperature_type: Cpu
    target_temperature_celsius: 75
    
cooling_policies:             # 调速策略：温度→转速的对应曲线
  - policy_idx: 0
    exp_cond_val: EnergySaving
    temperature_range_low: [20, 60, 70]
    speed_range_low: [30, 50, 80]
    
cooling_areas:                # 调速区域：哪些风扇受哪些策略控制
  - area_id: 1
    requirement_idx: 1        # 引用温度监控点
    policy_idx_group: [0, 1]  # 引用调速策略
    fan_idx_group: [1, 2, 3]  # 引用风扇
    
fan_types:                    # 风扇型号库：记录每种型号风扇的物理参数
cooling_fans:                 # 具体风扇实例：每个物理风扇一条
abnormal_fans:                # 异常处理规则：风扇故障时如何补偿
basic_cooling_config:         # 风扇分组和 PSU 校准参数
```

这些实体之间存在严格的引用关系：温度监控点 → 被调速区域引用 → 调速区域同时引用策略和风扇。删除任何一个实体，都可能导致引用悬空（就像删除了一个被引用的变量）。

#### 调速曲线是最核心的配置内容

调速策略里最关键的是**温度-转速曲线**：当温度是 X 时，风扇应该转 Y%。这条曲线不是直线，而是分段折线——温度低时转速增长缓慢，接近危险温度时急速拉高。

在 YAML 里，这条曲线用 4 个数组描述（低区间温度、高区间温度、低区间转速、高区间转速），可读性极差：

```yaml
temperature_range_low: [-127, 20, 40, 60]
temperature_range_high: [20, 40, 60, 127]
speed_range_low: [20, 30, 50, 80]
speed_range_high: [30, 50, 80, 100]
```

没有可视化的话，工程师根本看不出这条曲线的"形状"是否合理。

### 3.2 具体使用场景

#### 场景一：新机型适配——从零创建一套调速配置

一款新服务器型号即将量产，BMC 固件工程师需要为它创建完整的调速配置。这台机器有：
- 3 路温度传感器（CPU 进风口、CPU 核心、硬盘区域）
- 2 种工作模式（节能、高性能）
- 5 个物理风扇，分为前区（3个）和后区（2个）两个调速区域

手工编写这个 YAML，光是不同实体的引用关系（哪个区域引用哪些策略、哪些风扇）就极易出错。工具提供表单引导和关联选择，把"写 YAML"变成"填表格"。

#### 场景二：策略调优——线上机器噪音投诉

某型号服务器在数据中心运行时噪音过大，运维反馈风扇在 45°C 时就开始大幅拉转速，实际上 55°C 以下完全没必要。工程师需要调整节能模式的调速曲线，让转速在低温区域保持更平缓。

工具的可视化曲线编辑器让工程师直接拖拽折线节点，而不是手工修改 4 个数组的数值——修改完立即能看到新曲线的形状是否符合预期。

#### 场景三：配置交接——让新成员理解现有配置

一个已有的调速配置 YAML 有几百行，新加入的工程师需要理解它。工具的表单视图将这些字段用自然语言标注并分组展示，比直接啃 YAML 友好得多，是一个"配置可视化阅读器"。

| 场景 | 触发 | 核心操作 | 产出 |
|------|------|----------|------|
| 新机型适配 | 新硬件上线 | 逐实体填写表单，建立引用关系 | 完整 YAML 文件 + CSR 对象片段 |
| 策略调优 | 噪音/过热投诉 | 拖拽调速曲线，调整温度-转速映射 | 更新后的 YAML 和 CSR |
| 配置交接 | 新成员上手 | 用表单视图阅读现有 YAML | 理解现有配置逻辑 |
| 生成 CSR | 发布前检查 | 一键校验 + 生成 CSR 对象 | PSR/SR JSON 片段，注入固件配置 |

### 3.3 目标用户

| 用户类型 | 频率 | 核心诉求 |
|----------|------|----------|
| **BMC 固件工程师** | 高频，新机型迭代期 | 快速创建/修改，不犯引用错误 |
| **系统集成工程师** | 中频，集成阶段 | 验证配置有效性，生成正确的 CSR |
| **测试工程师** | 低频 | 构造边界用例（故障风扇、极端温度） |
| **新成员 / 接手者** | 一次性，上手期 | 用可视化方式理解复杂配置 |

### 3.4 用户痛点分析

#### 痛点①：YAML 手写的错误率极高，且错误隐蔽

调速配置 YAML 的字段众多（仅温度监控点就有 15+ 个字段），字段名拼写错误不会被 YAML 语法检查器发现（YAML 只检查格式，不检查字段名），运行时才会发现配置无效。

更危险的是类型错误：`fan_idx_group: 1` 和 `fan_idx_group: [1]` 在 YAML 里都合法，但前者是数字而后者是数组，程序处理时行为完全不同。

**痛点核心：YAML 格式允许的错误，程序可能要在运行时才能发现，而运行时就是在真实硬件上。**

#### 痛点②：实体间引用关系容易产生"悬空引用"

调速区域（CoolingArea）引用了温度监控点（requirement_idx: 3）和若干策略（policy_idx_group: [0, 1]）。如果工程师删除了策略 1，他必须记得同时去所有引用了策略 1 的调速区域里移除这个引用——遗漏就会导致配置错误，且不报错，只是静默失效。

当配置里有 5 个区域、6 个策略、8 个温度点时，手工维护引用关系几乎必然出错。

**痛点核心：多对多的引用关系无法靠人脑可靠追踪，遗漏不会立即报错。**

#### 痛点③：调速曲线对人不可读

温度-转速曲线被存储为 4 个平行数组，没有任何直观意义：

```yaml
temperature_range_low: [-127, 20, 40, 60]
temperature_range_high: [20, 40, 60, 127]
speed_range_low: [20, 30, 50, 80]
speed_range_high: [30, 50, 80, 100]
```

这条曲线在 60°C 以上会把风扇拉到 80~100%，但你从这串数字完全感知不到它的"形状"。工程师只能在脑子里点对点连线，还要注意 `_low` 和 `_high` 数组的对应关系……非常费力，且极易填错。

**痛点核心：配置的核心内容（曲线形状）对人脑是不透明的，工程师无法直觉判断曲线是否合理。**

#### 痛点④：生成 CSR 的转换规则复杂，手动转换不可接受

最终这份 YAML 需要被转换成 CSR 对象片段（两种格式：PSR 用于 PlatformServiceRegistry，SR 用于 ServiceRegistry）注入固件。这个转换过程有复杂的规则（字段重命名、数组展开、引用替换……），手动转换是不可接受的，但工程师之前只能依靠私下的脚本或口口相传的转换规则。

**痛点核心：配置到部署之间有一个"转换鸿沟"，需要专门的工具桥接。**

#### 痛点⑤：多台机型的配置管理混乱

不同服务器型号有各自的调速配置，但它们之间有大量相似结构。工程师往往用"复制 YAML 文件，手工修改部分字段"的方式管理多机型配置——容易改漏字段，也容易不知道两份配置之间有什么区别。

**痛点核心：多版本配置管理依赖文件系统 + 人工 diff，缺乏结构化的变更追踪。**

### 3.5 设计机会点

#### 🔴 高优先级

**A. 实体关系完全不可视化**
当前 6 类实体被平铺为独立的折叠区块，工程师无法直观看到"温度点→策略→区域→风扇"这条引用链。当配置规模增大（8 个温度点 × 5 个策略 × 4 个区域），全局关系极难追踪，引用关系只能靠读 ID 数字来对应。
> **建议**：增加"关系总览"面板——用节点图展示各实体间的引用关系，节点色区分实体类型；孤立节点（无任何引用/未被引用）标黄警告；点击节点跳转到对应表单区域。

**B. 调速曲线编辑器可发现性极低**
曲线编辑器是整个工具最有价值的功能，但它藏在每个策略卡的"编辑曲线"按钮后面，初次使用者几乎不会主动发现。且曲线编辑区域出现在页面底部，与对应策略卡在空间上脱离，上下文割裂。
> **建议**：在每个策略卡内嵌迷你曲线预览图（只读，约 120×60px），让用户直观感知曲线形态；点击后弹出模态框编辑，而非滚动到页面底部。

**C. 校验错误定位不可靠**
点击错误列表可尝试高亮对应字段，但因表单动态渲染，`querySelector` 有时找不到目标元素，体验断裂。
> **建议**：改用稳定的 `data-entity-id` + `data-field` 属性定位，高亮时同时展开该实体所在折叠区块、滚动到可视区，并在字段旁显示内联错误说明。

#### 🟡 中优先级

**D. 无配置预设/模板库**
每次新机型都从空表单开始，重复填写大量相同的结构（标准的 5 区域风扇配置、标准的三模式策略……）。
> **建议**：内置 2-3 个快速启动模板（"标准1U服务器"、"高密存储服务器"），选择后加载示例配置，在此基础上修改，而非从零填写。

**E. 长表单导航困难**
配置规模增大后，页面会非常长，用户难以定位目标实体（"第 3 个温度监控点在哪？"）。
> **建议**：左侧增加悬浮大纲导航，列出所有实体的 ID + 描述，点击锚点跳转；支持"折叠全部 / 展开全部"快捷操作。

**F. 生成 CSR 后无变更对比**
修改配置后重新生成 CSR，无法知道本次修改影响了哪些 CSR 字段，只能整体复制后靠外部 diff 工具对比。
> **建议**：在输出面板增加"与上次对比"功能，对两次生成结果做 JSON diff，增删字段用红/绿高亮标注。

**G. YAML 编辑器功能弱**
当前是普通 `<textarea>`，无语法高亮、无行号、无错误下划线，对习惯直接编辑 YAML 的资深工程师不友好。
> **建议**：引入轻量级代码编辑器组件（如 CodeMirror YAML 模式），提供语法高亮、行号、解析错误标红。

#### 🟢 低优先级

**H. 实体新增时默认值不完整**
新建实体时，许多必填字段留空，立即生成 CSR 时会报出一批错误，体验挫败。
> **建议**：新建实体时生成合理的占位默认值（如自动分配递增 ID、填入示例 CSR 路径格式），让用户在可工作的基础上修改。

**I. 多选下拉在小屏下被遮挡**
多选 Checkbox 浮层在视口底部区域会被截断。
> **建议**：检测展开方向，接近底部时向上弹出（Popover 方向自适应）。

## 4. 横向对比与共性机会点

### 4.1 三个工具的定位矩阵

|  | SMC 计算器 | 表达式计算器 | 调速配置模板 |
|--|-----------|------------|------------|
| **核心价值** | 消除手工位运算 | 缩短表达式调试周期 | 约束 + 可视化复杂配置 |
| **复杂度** | 低（单界面，2个方向） | 中（双模式，嵌套状态） | 高（6类实体，引用关系图） |
| **使用频率** | 高频、零散 | 中频、集中 | 低频、深度 |
| **错误代价** | 低（算错了重来） | 中（逻辑错误影响一类数据） | 高（配置错误影响真实硬件） |
| **用户最大痛点** | 手工移位出错 + 日志数字不可读 | 反馈周期长 + 中间状态不可见 | 引用关系维护难 + 曲线不可视 |
| **设计最大缺口** | 位图可视化 + 解码入口发现性 | 双模式割裂 + 失败 diff | 关系图 + 曲线编辑器发现性 |

### 4.2 三个工具共有的设计机会

**① 历史记录与会话恢复**
三个工具都没有持久化机制，刷新页面输入全部丢失。调试场景经常需要反复切换、对比不同版本的输入。
> 统一使用 `localStorage` 存储输入状态，页面加载时自动恢复上次会话；支持手动命名保存（"正式机型 A"、"测试用例集 v2"）。

**② 无快捷键支持**
专业工具用户对键盘效率有强烈期望，但三个视图均无快捷键。
> 为核心操作定义键盘快捷键（如 `Ctrl+Enter` = 执行/计算，`Ctrl+Shift+C` = 复制结果，`Escape` = 重置/关闭），并在操作按钮的 Tooltip 里展示。

**③ 深链接 / 分享**
工程师在 Review 或讨论 Bug 时希望把"当前填写的参数状态"分享给同事，但三个工具的状态都不体现在 URL 中。
> 将关键输入状态序列化为 URL QueryString，支持一键"复制分享链接"，对方打开链接即可看到同样的输入状态。

**④ 暗色主题硬编码**
三个工具都是硬编码的深色调色板，无法适配浅色系统主题，部分工程师在明亮环境下工作时会有可读性问题。
> 改用 CSS custom properties，监听 `prefers-color-scheme` 自动切换；或在全局 Tab 栏提供主题开关。

**⑤ 跨工具的上下文传递**
三个工具目前完全独立。但实际工作流中，工程师可能先用"SMC 计算器"得到一个偏移量，再把这个值作为"表达式计算器"的输入参数。工具之间没有数据传递通道。
> 长期机会：考虑在工具间提供"发送到 X 工具"的快捷操作，把当前计算结果预填到另一个工具的输入框中，打通工作流。
